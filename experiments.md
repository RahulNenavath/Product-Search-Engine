# Search Ablation Experiments

All experiments use the same evaluation setup unless noted:
- **Test set:** `Data/PROCESSED/test_queries.csv` — 200 queries (100 ESCI + 100 WANDS)
- **Qrels:** `Data/PROCESSED/test_qrels.json`
- **Retrieval pool (topn):** 100
- **Cutoffs (k):** 5, 10, 20
- **Embedding service:** `bge-reranker-v2-m3` cross-encoder + fine-tuned `all-MiniLM-L6-v2` encoder on GCP Cloud Run (NVIDIA L4 GPU)

---

## Experiment 1 — Baseline Strategies (Non-agentic)

**Source:** Prior full-corpus benchmark runs (9,100-query ablation for Hybrid Rerank; 200-query test set for others). Reported in README.

### NDCG@K

| Strategy | NDCG@5 | NDCG@10 | NDCG@20 |
|---|---|---|---|
| BM25 | 0.6030 | 0.5767 | 0.5530 |
| HNSW (Fine-tuned Encoder) | 0.6335 | 0.6149 | 0.6031 |
| Hybrid RRF (BM25 + HNSW, C=50) | 0.6576 | 0.6373 | 0.6176 |
| **Hybrid + Cross-Encoder Reranker (C=25)** | **0.6824** | **0.6608** | **0.6510** |

### MRR@K

| Strategy | MRR@5 | MRR@10 | MRR@20 |
|---|---|---|---|
| BM25 | 0.7598 | 0.7671 | 0.7689 |
| HNSW (Fine-tuned Encoder) | 0.8141 | 0.8176 | 0.8186 |
| Hybrid RRF | 0.8418 | 0.8466 | 0.8470 |
| **Hybrid + Cross-Encoder Reranker** | **0.8262** | **0.8305** | **0.8316** |

### Recall@K

| Strategy | Recall@5 | Recall@10 | Recall@20 |
|---|---|---|---|
| BM25 | 0.1544 | 0.2627 | 0.3896 |
| HNSW (Fine-tuned Encoder) | 0.1662 | 0.2951 | 0.4440 |
| Hybrid RRF | 0.1705 | 0.2976 | 0.4471 |
| **Hybrid + Cross-Encoder Reranker** | **0.1839** | **0.3151** | **0.4758** |

### Key Findings
- Each architectural stage adds value: BM25 → HNSW (+3.1pp NDCG@5), HNSW → Hybrid RRF (+2.4pp), Hybrid RRF → Cross-encoder (+2.4pp).
- Hybrid RRF has the best MRR (0.842) despite lower NDCG than the cross-encoder — the cross-encoder occasionally demotes the single most relevant result when the candidate pool (C=25) is tight.
- Candidate pool C=25 was found optimal for the cross-encoder from a 9,100-query ablation (larger pools feed more noise than signal).

## Experiment 2 — Agentic Search (LangGraph + Gemini Flash)

**Config:** LangGraph pipeline wrapping `hybrid_rerank`. `understand_query` model: `gemini-3.1-flash-lite-preview`. `assess_results` model: `gemini-3-flash-preview`. Candidate pool: 25. Simple queries skip `assess_results`.

**Source:** 200-query test set run.

| Metric | hybrid_rerank | Agentic (v1) | Δ |
|---|---|---|---|
| NDCG@5 | 0.682 | 0.657 | −3.7% |
| NDCG@10 | 0.661 | 0.635 | −3.9% |
| NDCG@20 | 0.651 | 0.628 | −3.5% |
| MRR@5 | 0.826 | 0.804 | −2.7% |
| Recall@20 | 0.476 | 0.459 | −3.6% |
| Latency | ~0.5s/q | ~3.7s/q | 7× slower |

### Key Findings
- Agentic query rewriting slightly underperforms the baseline at every metric.
- Root cause: the fine-tuned encoder + cross-encoder already handles vocabulary gaps well; LLM rewrites introduce marginal noise on the ~80% of well-specified queries.
- Latency is 7× higher with no NDCG gain — not viable for production without architectural changes.

## Experiment 3 — HyDE, LLM Reranker, and Combined Ablations

**Config:** All on 200-query test set. Candidate pool C=25 (matches cross-encoder baseline for fair comparison).
- **HyDE:** HNSW leg uses 50/50 average of raw query vector + hypothetical product title embedding (generated by `gemini-3.1-flash-lite-preview`). BM25 and cross-encoder unchanged.
- **Hybrid + LLM Reranker:** Standard BM25+HNSW hybrid retrieval. Cross-encoder replaced by `gemini-3.1-flash-lite-preview` single-pass permutation reranker (`[2] > [1] > [3]` format, all C=25 candidates in one prompt).
- **HyDE + LLM Reranker:** HyDE vector for HNSW + LLM permutation reranker (both changes).

### NDCG@K

| Strategy | NDCG@5 | NDCG@10 | NDCG@20 | vs Baseline |
|---|---|---|---|---|
| Hybrid + Cross-encoder *(baseline)* | 0.682 | 0.661 | 0.651 | — |
| HyDE + Cross-encoder | 0.687 | 0.664 | 0.655 | +0.7% |
| **Hybrid + LLM Reranker (Flash-lite, C=25)** | **0.730** | **0.712** | **0.689** | **+7.0%** |
| HyDE + LLM Reranker (Flash-lite, C=25) | 0.730 | 0.709 | 0.689 | +7.0% |

### MRR@K and Recall@K

| Strategy | MRR@5 | MRR@10 | MRR@20 | Recall@5 | Recall@10 | Recall@20 |
|---|---|---|---|---|---|---|
| Hybrid + Cross-encoder | 0.826 | 0.831 | 0.832 | 0.184 | 0.315 | 0.476 |
| HyDE + Cross-encoder | 0.831 | 0.836 | 0.837 | 0.186 | 0.317 | 0.479 |
| Hybrid + LLM Reranker (Flash-lite, C=25) | 0.840 | 0.844 | 0.844 | 0.192 | 0.336 | 0.496 |
| **HyDE + LLM Reranker (Flash-lite, C=25)** | **0.844** | **0.850** | **0.851** | **0.193** | **0.339** | **0.499** |

### Key Findings
1. **LLM Reranker is the standout result.** At +7.0% NDCG@5, it delivers the single largest improvement in the entire project — larger than any previous architectural upgrade. Gemini Flash-lite applies product-domain world knowledge (synonym understanding, intent reasoning) that the fine-tuned cross-encoder cannot.
2. **HyDE alone is marginal (+0.7% NDCG@5).** The fine-tuned encoder already has strong query-document alignment; the hypothetical product title adds little extra signal. HyDE's gains are strongest in zero-shot/unsupervised settings (as the paper notes) — not applicable here.
3. **HyDE + LLM Reranker ≈ Hybrid + LLM Reranker on NDCG, but wins on MRR.** NDCG@5 is identical (0.730); HyDE adds a marginal edge on MRR@5 (0.844 vs 0.840) and Recall@20 (0.499 vs 0.496). HyDE slightly improves the candidates entering the LLM reranker, surfacing the single best result at rank 1 more often.
4. **Agentic is the weakest LLM-backed approach.** Query rewriting (agentic) NDCG@5 = 0.657 vs permutation reranking NDCG@5 = 0.730. The LLM adds far more value *after* seeing the retrieved candidates than *before* retrieval.
5. **The balanced-50 test set was misleading.** On balanced-50 (25 garbled ESCI + 25 WANDS), the LLM reranker scored 0.156 vs cross-encoder 0.180 (−13%). On the full 200-query test, it scores 0.730 vs 0.682 (+7%). Garbled/navigational queries suppress LLM reranker gains — the full test set is the valid evaluation surface.

## Experiment 4 — LLM Reranker: Model and Pool Size Ablations

**Goal:** Isolate the effect of (a) upgrading the LLM from Flash-lite to Flash, and (b) increasing the candidate pool from 25 to 50, on the Hybrid + LLM Reranker strategy.

### Results

| Ablation | Model | Pool | NDCG@5 | NDCG@10 | NDCG@20 | MRR@5 | MRR@20 | Recall@20 | Latency | vs Flash-lite C=25 |
|---|---|---|---|---|---|---|---|---|---|---|
| Baseline | Flash-lite | 25 | 0.730 | 0.712 | 0.689 | 0.840 | 0.844 | 0.496 | ~2.7s/q | — |
| A | Flash | 25 | 0.739 | 0.725 | 0.699 | 0.853 | 0.858 | 0.503 | 16.4s/q | +1.2% NDCG@5 |
| B | Flash | 50 | 0.747 | 0.726 | 0.705 | 0.852 | 0.857 | 0.508 | 16.6s/q | +2.3% NDCG@5 |
| C | Flash-lite | 50 | 0.727 | 0.704 | 0.683 | 0.840 | 0.846 | 0.493 | 5.6s/q | −0.4% NDCG@5 |

### Ablation A Analysis

Upgrading from Flash-lite to Flash (same pool=25) improves NDCG@5 by **+1.2%** (0.730 → 0.739) and MRR@5 by **+1.5%** (0.840 → 0.853). Every metric improves consistently, confirming that the stronger reasoning model produces better permutation rankings. However, latency jumps from ~2.7s to **16.4s per query** — a **6× slowdown** — making Flash impractical for real-time use without caching. Worth pursuing for offline re-ranking or batch pipelines.

### Ablation B Analysis

Increasing the pool from 25 → 50 with Flash model gives another **+1.1% NDCG@5** (0.739 → 0.747) and **+0.9% NDCG@20** (0.699 → 0.705). Recall@20 improves modestly (+0.9%). Notably, **MRR is flat** (0.853 → 0.852) — the larger pool adds more relevant products deeper in the list (lifting NDCG) but Flash already surfaces the top-1 result correctly with pool=25 (so MRR doesn't benefit). Latency is essentially unchanged at 16.6s/q — the bottleneck is model response time, not prompt length.

### Ablation C Analysis

Flash-lite with pool=50 **regresses vs Flash-lite with pool=25**: NDCG@5 drops from 0.730 → 0.727 (−0.4%) and Recall@20 drops from 0.496 → 0.493 (−0.6%). MRR is flat. This is the opposite of what pool=50 did for Flash (+1.1%). The explanation is model capacity: Flash-lite lacks the reasoning ability to effectively rank 50 candidates in a single pass — the harder, longer prompt produces noisier permutations than it does with 25 candidates. Flash can leverage more candidates; Flash-lite cannot.

### Experiment 4 Summary — Key Findings

| Ablation | Model | Pool | NDCG@5 | vs Baseline | Latency |
|---|---|---|---|---|---|
| Baseline | Flash-lite | 25 | 0.730 | — | 2.7s/q |
| C | Flash-lite | 50 | 0.727 | −0.4% | 5.6s/q |
| A | Flash | 25 | 0.739 | +1.2% | 16.4s/q |
| **B** | **Flash** | **50** | **0.747** | **+2.3%** | 16.6s/q |

1. **Flash model consistently beats Flash-lite** regardless of pool size. Stronger reasoning = better permutation rankings.
2. **Larger pool helps Flash but hurts Flash-lite.** Flash can rank 50 candidates effectively; Flash-lite is overwhelmed and produces noisier orderings. Pool size is a model-capacity-dependent hyperparameter.
3. **Flash + pool=50 is the overall winner** (NDCG@5 = 0.747) but at 16.6s/query — only viable for offline/batch use.
4. **Flash-lite + pool=25 remains the best latency/quality tradeoff** (NDCG@5 = 0.730, 2.7s/query) for production real-time search.
